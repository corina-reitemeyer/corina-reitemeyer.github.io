Sprint 2 - CSS Concepts: Classes vs. IDs

An exploration by a newbie developer of when to use Classes and when to use ID's in CSS.

10 November 2010

When should you use a class, and when should you use an ID?

This is the type of question that sometimes still puzzles me when I make an attempt with HTML and CSS, but the more exemplars I find, the easier it is to understand why these two rules exist.

Generally it is said that classes are used for reusable styles, and ID’s are meant for unique elements. This means that classes are a lot more flexible, whereas ID aren’t and should be used sparingly.

But if you can apply styles to either one, when do you use which and in what scenario? For styling, I like to think of the question “how often does the styling repeat itself?”.

For example, look at this news video section on BBC.com. Each element looks the same. This is achieved, because someone created a class, added the styling for this particular class, and then used this class for each HTML element of each news video feature. This means the styling of this class has been used at least 5 times on this page alone.

Now imagine if they all used an ID. Of course, that would be possible and would work, but let me illustrate how much extra work and risk of maintenance and mistakes that can cause down the track.

Each element would have to have a unique ID to start with - 5 IDs to be exact. Then in the CSS you’d have to write (or copy & paste) the same styling for each ID separately. At the start, that may not look like a big issue because there are just 5 of them (if you’re lazy, that may be 4 too many already!). But imagine you want to change the styling in a years time. And you have 100 news video elements instead of just 5 of them.

Instead of having to change the styling in one class declaration in your CSS (which means you’re doing the work once), you suddenly have to change the styling all 5 or 100 of the IDs! That is not only tedious, but can also invite mistakes, which then would be tricky to find and and correct.

Wouldn’t it be a lot better if you just had one class to debug and maintain? Hence why they used a class instead of several IDs. Now, after looking via the Devtools, they have used ID’s as well, however my guess is it is for their javascript component.

This doesn’t mean ID’s are completely useless.

ID’s are handy if you are trying to create anchor-links. If you tried to link a menu item to a class on your site…how would the browser now where to send you if you had 100 elements with the same class? Exactly it would be confused. So an ID is appropriate here. Another use, as mentioned above, is Javascript in order to add interactivity. The ID selector is used by JavaScript to access and manipulate the element with the specific ID. Javascript has specific syntaxes that looks for ID’s, and hence using an HTML ID attribute makes sense here.
